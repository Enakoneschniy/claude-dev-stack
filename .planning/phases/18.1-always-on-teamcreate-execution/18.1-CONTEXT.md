# Phase 18.1: Always-on TeamCreate Execution - Context

**Gathered:** 2026-04-13
**Status:** Ready for planning

<domain>
## Phase Boundary

Replace the current "detect parallel → offer TeamCreate" model with an "always TeamCreate" model. When executing a milestone with multiple phases, ALL phase members are spawned upfront with task dependencies matching ROADMAP `depends_on`. No separate parallel/sequential code paths. Members auto-start when their blockers complete, idle when blocked.

This is an improvement to INFRA-04 (Phase 13) — same mechanism, different triggering model.

</domain>

<decisions>
## Implementation Decisions

### Execution Model

- **D-01:** TeamCreate is spawned at milestone-level, not per-phase. New entry point: either a `/gsd-execute-milestone` command or modification to transition.md Route A that spawns remaining phases as a team instead of offering one-at-a-time.
- **D-02:** One team member per phase. Member names match phase slugs (e.g., `phase-16-git-conventions`).
- **D-03:** Task dependencies mapped from ROADMAP.md `depends_on` → TaskCreate `blockedBy`. Phase 16 depends on 14 → task 16 blocked by task 14.
- **D-04:** Members with no blockers (or all blockers complete) start immediately. Members with pending blockers idle until notified.
- **D-05:** Each member runs full execute-phase flow: create branch, execute plans in waves, verify, report completion.
- **D-06:** Worktree isolation per member — same as current TeamCreate usage. Each member gets `isolation: "worktree"`.

### Error Handling

- **D-07:** If one member fails, other non-dependent members continue. Dependent members receive failure notification and either wait for retry or skip.
- **D-08:** Partial success is valid. Report which phases completed, which failed, which were never started (blocked on failed phase).
- **D-09:** User can retry failed phases individually after team completes.

### Integration Points

- **D-10:** Modify `transition.md` Route A to replace the current parallel detection block with always-team spawning. Remove the independence test — it's no longer needed when tasks have explicit blockedBy.
- **D-11:** Keep existing sequential fallback — if TeamCreate is unavailable (older Claude Code, restricted env), fall back to one-at-a-time execution as before.
- **D-12:** The team lead (orchestrator) monitors task completion and sends messages to unblock waiting members when their dependencies complete.

### Scope Limits

- **D-13:** This phase modifies GSD workflow files only (`transition.md`). No changes to the npm package source code.
- **D-14:** No new commands needed — the change is transparent to users. They still run `/gsd-execute-phase N` and transition handles the rest.

### Claude's Discretion

- Exact TaskCreate/blockedBy wiring syntax
- How to detect "all remaining phases" from ROADMAP.md (reuse `roadmap analyze`)
- Whether team lead should also be a member or just coordinate

</decisions>
