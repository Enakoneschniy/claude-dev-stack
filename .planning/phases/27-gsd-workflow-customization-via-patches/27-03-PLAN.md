       ```markdown
       ## GSD Local Patches (claude-dev-stack)

       Problem: Get-Shit-Done (GSD) installs workflows globally under
       `~/.claude/get-shit-done/workflows/`. Any local edit there is wiped by
       `/gsd-update`. Projects that need different workflow behavior
       (e.g., a different transition agent, a customized manager prompt) lose
       their changes on every update.

       Solution: A package-shipped `patches/` directory + a SessionStart hook
       that re-applies the patches whenever the installed workflow files drift
       from the shipped patch copies. Formalized in claude-dev-stack Phase 27
       (requirement GSD-01).

       ### Layout

       In the claude-dev-stack repo:

       - `patches/manager.md`, `patches/transition.md` — full-file
         replacements. Each `patches/<name>.md` overrides
         `~/.claude/get-shit-done/workflows/<name>.md`.
       - `hooks/gsd-auto-reapply-patches.sh` — SessionStart hook. Iterates
         every `*.md` in the resolved patches dir; for each, if the target
         workflow file has a different SHA-256, copies the patch over it.
       - `lib/install/hooks.mjs` — install wizard copies the shipped
         `patches/` tree to `~/.claude/gsd-local-patches/` so the hook has a
         version-pinned, wizard-authoritative source that survives
         `/gsd-update`.

       ### Resolution precedence (BUG-06 D-07)

       The hook picks the first directory that exists, in this order:

       1. `$PATCHES_DIR` env var (test/dev override).
       2. `~/.claude/gsd-local-patches/` — wizard-pinned (authoritative).
       3. npm global — `$(npm root -g)/claude-dev-stack/patches` or
          `$HOME/.npm/_npx/*/node_modules/claude-dev-stack/patches`.
       4. Well-known dev checkouts — `$HOME/Projects/claude-dev-stack/patches`,
          etc.

       ### Interaction with `/gsd-update`

       - `/gsd-update` rewrites `~/.claude/get-shit-done/workflows/` with the
         upstream GSD defaults.
       - On the next Claude Code session, the SessionStart hook runs and
         re-copies any patch whose SHA-256 differs from the upstream file,
         restoring the local customization.
       - Users see `GSD patches auto-reapplied (N file(s) updated)` on start
         when a patch was restored; silent otherwise.

       ### When upstream GSD changes the patched workflow

       If upstream ships a new version of a workflow you've patched (e.g., a
       new `manager.md`), your patch goes stale and silently replaces the new
       upstream behavior with the old patched one. Mitigations:

       - Re-merge the patch against the new upstream: copy
         `~/.claude/get-shit-done/workflows/<name>.md` (the fresh upstream) to
         a scratch dir, re-apply your customization on top, then save back to
         the claude-dev-stack `patches/<name>.md`. Commit with a note like
         `fix(patches): re-merge manager.md against GSD v<version>`.
       - `/gsd-reapply-patches` skill can assist.

       ### Testing

       Two regression tests under `tests/` lock the mechanism:

       - `tests/gsd-auto-reapply-patches.test.mjs` — SHA-diff apply,
         idempotency, precedence (pinned > dev checkout), missing-GSD exit,
         missing-source exit.
       - `tests/install-patches-copy.test.mjs` — wizard copies
         `patches/*.md` to `~/.claude/gsd-local-patches/`.

       ### References

       - claude-dev-stack ROADMAP Phase 27 (GSD-01)
       - `.planning/phases/27-gsd-workflow-customization-via-patches/`
       - BUG-06 D-07 decision in Phase 19
       ```
