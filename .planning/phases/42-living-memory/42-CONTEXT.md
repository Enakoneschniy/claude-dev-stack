# Phase 42 Context

**Phase:** 42 — Living Memory
**Mode:** auto-discussion
**Created:** 2026-04-17

## Goal

Make claude-dev-stack's SQLite session memory **actively useful** — not just stored, but automatically loaded, queried, and surfaced during Claude Code sessions. Users should never need to explicitly search for past decisions — Claude remembers by default.

## Prior state

Phase 37 shipped 5 MCP tools: `sessions.search`, `sessions.timeline`, `sessions.get_observations`, `docs.search`, `planning.status`. They work (tested via MCP stdio integration test). But:
- **Nobody calls them** — CLAUDE.md doesn't instruct Claude to use them
- **SessionStart hook** only reads markdown `context.md` — doesn't query SQLite
- **No CLI commands** for direct user access (`claude-dev-stack search/stats`)
- **No skills** wrapping MCP tools for easy Claude Code invocation

## Decisions

### D-140: SessionStart SQLite context injection

**Decision:** Extend `hooks/session-start-context.sh` to call `claude-dev-stack memory` CLI subcommand that queries `sessions.timeline` for the last 3 sessions and prints a condensed summary of key observations (decisions, bugs, patterns). This output gets injected alongside the existing `context.md` content.

**Format:**
```
📋 Recent memory (from SQLite):
Session 2026-04-17: [3 observations] — Phase 41 UAT, Docker devcontainer, /cds-quick rewrite
Session 2026-04-16: [5 observations] — Phase 39 execute, Phase 40 plan+execute
Session 2026-04-15: [2 observations] — Phase 38 backfill migration

Use sessions.search MCP tool for deeper queries.
```

**Why a CLI subcommand (not inline node):** keeps the hook shell-only (POSIX compatible), delegates heavy lifting to the bundled JS. The `memory` command reads `sessions.db` directly (no MCP round-trip at hook time).

### D-141: CLAUDE.md template — auto-use MCP tools

**Decision:** Update the CLAUDE.md template (generated by wizard via `lib/install/claude-md.mjs`) to include a "Memory" section instructing Claude to:
- Call `sessions.search` when user asks about past decisions/work ("что мы решили", "when did we", "как мы делали")
- Call `docs.search` when user asks about project documentation
- Call `planning.status` when user asks about project progress
- Never say "I don't have access to past sessions" — always try the MCP tool first

**Placement:** after "Session Protocol" section in CLAUDE.md.

### D-142: `/cds-search` skill

**Decision:** Skill that wraps `sessions.search` MCP tool. User types `/cds-search "тема"` → Claude calls `mcp__cds__sessions.search({query: "тема"})` → formats results as a readable list.

**Behavior:** If MCP tool not available (server not registered), falls back to `claude-dev-stack search "тема"` via Bash.

### D-143: `/cds-stats` skill

**Decision:** Skill that shows project memory stats. Calls `mcp__cds__planning.status` + counts sessions/observations from `sessions.timeline`. Outputs:
- Session count (total + last 7 days)
- Observation count by type (decision, bug, pattern, etc.)
- Top entities (most referenced)
- Last activity timestamp

### D-144: `claude-dev-stack search "query"` CLI

**Decision:** New CLI subcommand. Calls `sessions.search` directly (imports from `@cds/core` or reads `sessions.db`). Outputs results as formatted text to stdout. Works outside Claude Code (standalone terminal).

### D-145: `claude-dev-stack stats` CLI

**Decision:** New CLI subcommand. Reads `sessions.db` directly, prints dashboard:
```
📊 Project: claude-dev-stack
Sessions: 42 (7 this week)
Observations: 156 (decisions: 34, bugs: 12, patterns: 28, other: 82)
Entities: 89 (top: SQLite, dispatchAgent, Phase 39, Claude Code, tsup)
Last activity: 2026-04-17
```

### D-146: `claude-dev-stack memory` CLI (SessionStart helper)

**Decision:** Internal CLI subcommand called by SessionStart hook. Reads last N sessions from `sessions.db`, prints condensed summary to stdout (which gets injected into Claude's context). Not user-facing — no help entry.
