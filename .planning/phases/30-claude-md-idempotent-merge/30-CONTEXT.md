# Phase 30: CLAUDE.md Idempotent Merge — Context

**Gathered:** 2026-04-15
**Status:** Ready for planning

<domain>
## Phase Boundary

Wizard preserves user-written CLAUDE.md content across runs. Our claude-dev-stack instructions live in a delimited section between markers; re-install only updates that section, never touches user content.

Scope:
- BUG-07 (already in REQUIREMENTS.md)
- `generateClaudeMD()` in `lib/install/claude-md.mjs` rewrites to delegate to `updateProjectClaudeMd()` (or equivalent marker-based function) in `lib/project-setup.mjs`
- Expand `generateSkillsSection()` → `generateClaudeDevStackSection()` so the managed block includes Language, Auto-Routing, Knowledge Base, Session Protocol, Code Style, Rules, References + Skills

Already exists and NOT re-done:
- `lib/project-setup.mjs` — `MARKER_START`/`MARKER_END` constants + `updateProjectClaudeMd()` function with 3-path logic (create/update/append)
- Existing 3-case handling: new file, markers present → replace between, markers absent → append

</domain>

<decisions>
## Implementation Decisions

### Delegate to existing marker merge (SC#1..6)
- **D-01:** Rewrite `generateClaudeMD()` in `lib/install/claude-md.mjs` to build its full managed-section content (Language, Auto-Routing, Knowledge Base, Session Protocol, Code Style, Rules, References, Skills subsection) and pass it to a new function `updateManagedSection(projectPath, managedContent)` exported from `lib/project-setup.mjs`.
- **D-02:** `updateManagedSection()` is a minor refactor of `updateProjectClaudeMd()`: takes arbitrary managed content (not only skills section), applies same 3-path logic with same `<!-- @claude-dev-stack:start -->` / `<!-- @claude-dev-stack:end -->` markers.
- **D-03:** `updateProjectClaudeMd()` remains as a convenience wrapper that builds the skills section and calls `updateManagedSection()`. Backward compat preserved — any existing callers of `updateProjectClaudeMd()` keep working.

### Managed section content (SC#4)
- **D-04:** The managed content generated by `generateClaudeMD()` includes ALL current template sections (Language, Auto-Routing, Knowledge Base, Session Protocol, Code Style, Rules, GSD Multi-Phase Workflow, References, Skills). Anything previously hardcoded as full-file template now lives strictly between the markers.
- **D-05:** The Skills subsection is nested under the main managed block (single marker pair for entire claude-dev-stack content, no nested markers).

### Wizard status output (SC#7)
- **D-06:** Wizard step prints one of three lines based on `updateManagedSection()` return value:
  - `"CLAUDE.md: created"` (new file)
  - `"CLAUDE.md: updated"` (markers existed, content between markers replaced)
  - `"CLAUDE.md: appended"` (file existed without markers, managed section added to end)
  - NEVER `"CLAUDE.md: overwritten"` (that's the bug being fixed).

### Backwards compatibility (SC#6)
- **D-07:** For users who already have CLAUDE.md created by the buggy `writeFileSync` path: running the new wizard once will produce the `"appended"` path because their file has no markers. The appended section will be wrapped in markers. Subsequent re-installs will use the `"updated"` path. No data loss.
- **D-08:** Template file at `{vault}/CLAUDE.md.template` (currently written by wizard) should NOT use the marker-based approach — it's a reference file meant to be fully overwritten each install. Only the project-level CLAUDE.md is idempotent.

### Claude's Discretion
- Exact function name: `updateManagedSection()` vs `writeClaudeDevStackSection()` vs other
- Whether to keep both `updateProjectClaudeMd()` (skills-only) and new function, or deprecate the former
- Exact test harness: temp dir + multiple fixture scenarios (no CLAUDE.md, with markers, without markers, with user content before/after, etc.)

</decisions>

<specifics>
## Specific Ideas

- The wizard must print the status line (created/updated/appended) on a line by itself so users clearly see what happened. Color it green for "updated" (safe), yellow for "appended" (user content preserved but new markers added), blue for "created" (fresh install).
- Test cases MUST cover: (a) no CLAUDE.md → created, (b) existing CLAUDE.md with markers → updated in place, (c) existing CLAUDE.md without markers → appended at end, (d) existing CLAUDE.md with our markers + user content OUTSIDE markers → user content untouched, (e) re-running wizard twice is idempotent (second run = no diff).
- The managed section content depends on `profile` (language), `vaultPath`, `projectsData`, `skillsDir` — same inputs as current `generateClaudeMD()`. Pure function of those inputs.
- This phase depends on Phase 28 ONLY if Phase 28 updates the CLAUDE.md template string. If Phase 28 ships first, its new "Knowledge Base (Auto-Loaded)" text goes into the managed section. If Phase 30 ships first, Phase 28's text-change still works because it's inside the managed block.

</specifics>
