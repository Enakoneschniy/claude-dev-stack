<decisions>
## Implementation Decisions

### Node Version Baseline (D-33 … D-35)
- **D-33:** CDS baseline bumps to **Node 20+** for v1.0. This is a breaking change from v0.12.x Node 18 baseline. Justification: (a) Node 18 is EOL (2025-04-30 per Node release schedule), (b) `better-sqlite3@^12.9.0` (latest, actively maintained) requires Node 20+, (c) v1.0 alpha ships on `--tag alpha` (does NOT replace `@latest`), so existing v0.12.x users on Node 18 are not forced to upgrade until they opt into alpha.
- **D-34:** Use `better-sqlite3@^12.9.0` (latest stable as of 2026-04-16). Engines: `node: 20.x || 22.x || 23.x || 24.x || 25.x`. License: MIT. Prebuilds available via `prebuild-install` for common (darwin/linux/win32) × (x64/arm64) matrices — PROJECT.md "no post-install compilation" constraint is satisfied for common platforms. Users on exotic platforms (musl libc, uncommon archs) fall back to `node-gyp rebuild` which requires a C++ toolchain.
- **D-35:** Migration path for Node 18 users is documented in `docs/migration-v0-to-v1-alpha.md` (Phase 39 deliverable). Phase 35 only makes the baseline change; Phase 39 documents it.

### Schema Migration Mechanics (D-36 … D-39)
- **D-36:** Migrations are **numbered SQL files** in `packages/cds-core/src/vault/migrations/`: `001-initial.sql`, `002-*.sql`, etc. Sequential integer prefix, zero-padded to 3 digits. File content is pure SQL (DDL + optional DML). Bundler/TS compilation emits `dist/vault/migrations/*.sql` verbatim (via package.json `files` or a post-build copy step).
- **D-37:** **Auto-migrate on `openSessionsDB(projectPath)`**. Flow inside a single transaction:
  1. Ensure `schema_version` table exists (`CREATE TABLE IF NOT EXISTS schema_version (version INTEGER PRIMARY KEY, applied_at TEXT)`).
  2. Read current `MAX(version)`.
  3. Scan `migrations/*.sql`, filter to pending versions (> current).
  4. Execute each pending migration's SQL in order.
  5. Insert `(version, ISO timestamp)` into `schema_version` per migration applied.
  6. Commit transaction. If any step fails, rollback.
- **D-38:** No rollback/down migrations. Forward-only per REQ VAULT-02. If a migration is bad, the fix is another forward migration that corrects the state.
- **D-39:** The migration runner reads SQL files at runtime via `fs.readFileSync` relative to the compiled `dist/vault/migrations/` directory. This requires the `.sql` files to ship in the npm tarball (or be inlined at build time). We ship them via `packages/cds-core/package.json` `"files": ["dist"]` — `dist/vault/migrations/*.sql` is part of `dist`, so it ships automatically. No bundler inlining step required in Phase 35.

### VAULT-03 Boundary Enforcement (D-40 … D-42)
- **D-40:** Implementation = **folder convention + no re-export**:
  ```
  packages/cds-core/src/vault/
    internal/
      db.ts              ← openRawDb(), SessionsDB type, low-level helpers
  ```

  - Verifies `import('@cds/core').then(m => m.openRawDb)` is `undefined`.
  - Verifies `import('@cds/core/vault/internal/db')` either fails type-check OR throws at runtime (depending on `exports` map — we don't need a strict `exports` map since the package is private, but we document the boundary intention).

### Schema Structure (D-43 … D-47)
- **D-43:** Canonical tables (per REQ VAULT-02):
  - `sessions(id TEXT PRIMARY KEY, start_time TEXT, end_time TEXT, project TEXT, summary TEXT)`
  - `observations(id INTEGER PRIMARY KEY AUTOINCREMENT, session_id TEXT REFERENCES sessions(id), type TEXT, content TEXT, entities TEXT, created_at TEXT)` — `entities` is JSON
  - `entities(id INTEGER PRIMARY KEY AUTOINCREMENT, name TEXT UNIQUE, type TEXT, first_seen TEXT, last_updated TEXT)`
  - `relations(from_entity INTEGER REFERENCES entities(id), to_entity INTEGER REFERENCES entities(id), relation_type TEXT, observed_in_session TEXT REFERENCES sessions(id))`
- **D-44:** `observations.entities` JSON format = **array of integer entity IDs**: `[1, 5, 42]`. NOT inline entity copies. Entity names fetched via JOIN when needed for FTS5 matching or display. Keeps row size small, avoids duplication, aligns with the separate `entities` table.
- **D-45:** `entities.type` and `relations.relation_type` are **open strings**, not TS enums. Phase 36 Haiku extraction invents new types (`person`, `project`, `concept`, `decision`, `file`, `commit`, `skill`, `api`, ...) without schema migration. `@cds/core` exports a `CANONICAL_ENTITY_TYPES` array as a suggestion / autocomplete hint, but the DB accepts any string.
- **D-46:** FTS5 virtual table: `observations_fts USING fts5(content, session_summary, content=observations, content_rowid=id)` — external-content table indexing `observations.content` + a denormalized `session_summary` column (synced via triggers). Why denormalize: FTS5 cannot JOIN at query time; pre-joining session summary into the FTS row lets `sessions.search(query)` return ranked observations with context in one query.
- **D-47:** Triggers on `observations` INSERT/UPDATE/DELETE keep `observations_fts` in sync. Session summary sync: when `sessions.summary` updates, a trigger re-denormalizes into matching FTS rows.

### Path Resolution & DB Lifecycle (D-48 … D-50)
- **D-48:** Project path resolution: `openSessionsDB(projectPath)` accepts an absolute path like `/Users/x/Projects/claude-dev-stack`. The function computes `sessionsDbPath = path.join(os.homedir(), 'vault', 'projects', path.basename(projectPath), 'sessions.db')` using `path.basename` as the slug. If `~/vault/projects/{name}/` doesn't exist, it is created with `recursive: true`. Reusing existing CDS `reverseProjectMap()` is deferred — Phase 35 uses basename-only to minimize coupling to `lib/projects.mjs`.
- **D-49:** `openSessionsDB` **caches** the opened `SessionsDB` per `projectPath` in a module-level `Map`. Re-opening the same project returns the cached handle. Users can call `closeSessionsDB(projectPath)` explicitly; no automatic close-on-process-exit handler in Phase 35 (Node's default cleans up on exit).
- **D-50:** Concurrent access: WAL mode allows multiple readers + one writer. If two writers race (Stop hook + backfill migrator), the second one blocks briefly (~ms) — acceptable. No explicit lock API in Phase 35.

<specifics>
## Specific Ideas

- The Node 20+ bump is the largest user-facing consequence of v1.0. It cascades into Phase 33 Plan 04 CI matrix amend + root package.json engines update + Phase 39 migration guide.
- FTS5 external-content + denormalized session_summary is the right tradeoff: query simplicity > storage efficiency, sessions.summary is short (~1KB).
- Open-string entity types are deliberately non-defensive: LLMs (Haiku in Phase 36) will invent types we didn't predict; TS enum would force schema migration every time.
- Module-level cache in D-49 is fine for single-process usage (Stop hook, CLI command, backfill). For multi-process (parallel Claude Code sessions on same project), WAL mode handles it — no need for cross-process coordination in Phase 35.

</specifics>
