```javascript
#!/usr/bin/env node
/**
 * Thin wrapper around lib/session-context.mjs::updateContextHistory.
 *
 * Invoked from:
 *   1. hooks/session-end-check.sh (Stop hook safety net)
 *   2. skills/session-manager/SKILL.md /end code block (primary)
 *
 * Environment:
 *   VAULT_PATH          - vault root (default: $HOME/vault)
 *   CDS_PROJECT_NAME    - project slug under vault/projects/ (required)
 *
 * Args:
 *   argv[2]             - session log filename (required, e.g. "2026-04-10-fix-bug.md")
 *   argv[3]             - optional session title override
 *
 * Exit codes (D-14):
 *   0 — success OR non-fatal skip (vault missing, project missing, etc.)
 *   2 — programmer error (required args missing or invalid)
 *
 * Never propagates errors to the user's terminal. stderr is used for diagnostic messages
 * but the process never exits non-zero on filesystem conditions.
 */

import { homedir } from 'os';
import { dirname, resolve } from 'path';
import { fileURLToPath } from 'url';

import { updateContextHistory } from '../lib/session-context.mjs';

const __dirname = dirname(fileURLToPath(import.meta.url));

function main() {
  const vaultPath = process.env.VAULT_PATH || `${homedir()}/vault`;
  const projectName = process.env.CDS_PROJECT_NAME;
  const sessionLogFilename = process.argv[2];
  const sessionTitle = process.argv[3]; // optional

  if (!projectName) {
    process.stderr.write('update-context: CDS_PROJECT_NAME env var is required\n');
    process.exit(2);
  }
  if (!sessionLogFilename) {
    process.stderr.write('update-context: session log filename argument is required\n');
    process.exit(2);
  }

  try {
    const result = updateContextHistory({
      vaultPath,
      projectName,
      sessionLogFilename,
      sessionTitle,
    });
    // Diagnostic line on stderr (kept silent on hook invocation via 2>/dev/null)
    process.stderr.write(`update-context: ${result.action} (${result.entriesCount} entries)\n`);
    process.exit(0);
  } catch (err) {
    // Programmer errors (missing args, bad projectName) exit 2.
    // Anything else is a bug — still exit 0 to keep the hook quiet (D-14).
    const msg = err && err.message ? err.message : String(err);
    if (/required|projectName/.test(msg)) {
      process.stderr.write(`update-context: ${msg}\n`);
      process.exit(2);
    }
    process.stderr.write(`update-context: unexpected error: ${msg}\n`);
    process.exit(0);
  }
}

main();
```

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| shell → node wrapper | `hooks/session-end-check.sh` invokes the wrapper with env vars and a filename argument |
| node wrapper → helper | `hooks/update-context.mjs` calls `updateContextHistory` with caller-supplied values |
| skill bash block → node wrapper | `skills/session-manager/SKILL.md` /end block invokes the wrapper from inside a user shell session |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-02-01 | Injection | Session log filename argument | mitigate | The filename is derived inside the hook from `basename` of a path matched by `ls "$SESSION_DIR/$TODAY"*.md`, so it is always a plain filename. The wrapper passes it as a single argv element to `updateContextHistory`, which uses it only with `path.join` — no shell interpolation. No quoting pitfalls because no `eval`/`sh -c` is involved. |
| T-02-02 | Injection | `PROJECT_NAME` shell variable in the hook | mitigate | Passed via env var `CDS_PROJECT_NAME` to the node subprocess. The helper validates it with the path-traversal guard (T-01-03 from Plan 01). Even a maliciously crafted basename containing `/` or `..` throws a programmer error and the hook's `|| true` catches it silently. |
| T-02-03 | Denial of Service | Hook noise on node subprocess failure | mitigate | The node invocation uses `2>/dev/null` to suppress stderr AND `|| true` to ensure bash continues to the git push step even if node exits non-zero. D-14 + belt-and-suspenders. |
| T-02-04 | Tampering | Stop hook running context.md update BEFORE git push | mitigate | Ordering is enforced by the hook: context.md update happens inside the "session logged" branch, BEFORE the `git -C "$VAULT" add -A` line. So the updated content is staged and committed as part of the same Session commit. Verified by the `awk` acceptance criterion in Task 2. |
| T-02-05 | Elevation of Privilege | Skill block running as user shell | accept | The skill's /end block is a bash heredoc executed in the user's own shell. It inherits user privileges by design. No new attack surface beyond what the skill already had before this plan. |
| T-02-06 | Information Disclosure | `git rev-parse` in skill block on non-git cwd | mitigate | The skill falls back to `pwd` if `git rev-parse --show-toplevel` fails (existing behavior). The wrapper tolerates a non-existent vault path (returns `skipped`). No information leaks via error paths because stderr is suppressed by `2>/dev/null`. |
| T-02-07 | Repudiation | Missing audit trail for hook-triggered updates | accept | The vault's git history captures every context.md mutation as part of the session commit. That is the audit trail. No separate log needed. |

No `high` severity findings. All threats mitigated in code or accepted with rationale.
